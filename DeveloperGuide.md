# Developer Guide

## Welcome

Welcome to a project by **pastMidnight Development**.

Every project in this repository is built with a simple philosophy:

> **Build software that solves real problems, is easy to understand, and can be improved one step at a time.**

Many of these projects begin as tools to solve problems I encounter in my own work or hobbies. Rather than trying to build everything at once, I focus on creating small, working improvements that can be tested, refined, and expanded over time.

My goal is not simply to write code, it is to create software that remains understandable months or years later.

---

Development Motto

Build it. Test it. Improve it. Repeat.

---

# Development Philosophy

Every project follows these core principles:

* Modular design over large monolithic files.
* Readable code over clever code.
* Continuous improvement through small iterations.
* Consistent project structure across repositories.
* Clear documentation for future development.
* Build features only when they can be tested.
* Solve real-world problems before adding convenience features.

If a feature cannot be tested or does not improve the user experience, it waits until a future version.

---

# Coding Standards

Every source file includes:

* File name
* Version number
* Purpose

Standard file layout:

1. IMPORTS
2. CONFIGURATION
3. CONSTANTS
4. STATE
5. DATABASE (when applicable)
6. FUNCTIONS
7. MAIN
8. PROGRAM START

This consistent structure makes projects easier to navigate and maintain.

---

# Development Workflow

Every development session should end with a working application.

Rather than making hundreds of changes before testing, development is broken into small milestones that can be verified individually.

Example:

Session 1

* Database

Session 2

* Camera

Session 3

* Search

Each milestone should leave the project in a functional state.

# Repository Standards

Each project should include:

* README.md
* CHANGELOG.md

Documentation is treated as part of the project—not an afterthought.
